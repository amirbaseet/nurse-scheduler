Architecture

Both algorithms share the same input (AlgorithmConfig) and output (ScheduleResult).

Divergent Core Logic

V1 (Layer 4)V2 (nurseFirstAssign + clinicGapFill)
Strategy3-pass: specialist auto → affinity pre-assign → difficulty queue with MCV1-pass: constrained nurse ordering → best slot per day → gap fill
IterationIterates over clinic slotsIterates over nurses
OrderingMCV heuristic (hardest slots first)Most constrained nurse first (fewest available days)
BacktrackingYes (via tryBacktrack)Yes (in gap-fill phase)
Look-aheadYes (5 future slots)No (relies on constraint ordering)

Execution Flow

V1: Clinic-First Flow

Layer 1 → Block unavailable days
Layer 2 → Place fixed assignments
Layer 3 → Fill gender-restricted slots
Layer 4 → PRIMARY ASSIGNMENT (clinic-first):
  ├── Pass 0: Specialist auto-assign (P > 0.85)
  ├── Pass 1: Global affinity pre-assign (nurse's #1 clinic per day)
  ├── Pass 2: Confidence fast path (P > 0.50)
  └── Pass 3: Difficulty queue + MCV + scoring + look-ahead + backtrack
Layer 5 → Stack secondary clinics
Layer 6 → Patient call programs
Layer 7 → Fill remaining budget hours
Layer 8 → Mark remaining as OFF
Layer 9 → Simulated annealing optimization

V2: Nurse-First Flow

Layer 1 → Block unavailable days
Layer 2 → Place fixed assignments
Layer 3 → Fill gender-restricted slots
NurseFirstAssign → PRIMARY ASSIGNMENT (nurse-first):
  ├── Sort nurses by constraint level (fewest available days first)
  ├── For each nurse:
  │   ├── Calculate dailyTarget = remainingHours / availableDays
  │   ├── Sort available days by strongest clinic affinity
  │   └── For each day: score unfilled slots, assign best match
ClinicGapFill → Fill remaining clinic demands:
  ├── Build difficulty queue (hardest unfilled slots first)
  ├── For each unfilled slot: find candidates, score, assign best
  └── Backtrack if no direct candidates available
Layer 5 → Stack secondary clinics
Layer 6 → Patient call programs
Layer 7 → Fill remaining budget hours
Layer 8 → Mark remaining as OFF
Layer 9 → Simulated annealing optimization

Answers: "How good is this clinic slot for this nurse?"

ComponentRangeDescription
S_affinity0–400Historical probability + top-choice bonus
S_hours0–300How close shift hours match daily target

Key Differences

  1. S_hours (V2 only): Scores how close shiftHours matches dailyTarget. Perfect match = 300, within 30min = 250, within 1h = 200, within 2h = 100, poor = 30.
  2. S_variety (V2 only): Encourages clinic diversity. Starts at 100, loses 25 per same-clinic repeat day.
  3. S_demand (V2 only): Steers nurses toward high-need clinics. urgency = remaining / nursesNeeded * 200.
  4. S_lookahead (V1 only): Checks impact on 5 future slots. V2 processes nurses by constraint level instead.
  5. S_budget (V1 only): Uses budget ratio to prefer more-available nurses. V2 handles this via dailyTarget.
  6. S_fair (V1 only): Uses fairness ratio. V2 handles fairness by processing most-constrained nurses first.

Algorithm Selection (Router)

src/algorithm/router.ts
  └── runScheduleAlgorithm(version, weekStart, config)
        ├── "v1-clinic-first" → generateWeeklySchedule()   [src/algorithm/index.ts]
        └── "v2-nurse-first"  → generateWeeklyScheduleV2() [src/algorithm/v2/index.ts]

The router is called from src/app/api/schedule/generate/route.ts. The algorithmVersion is:

  • Received in the POST body (optional, defaults to v1-clinic-first)
  • Stored in WeeklySchedule.algorithmVersion column
  • Selected by the manager via a dropdown in the generate wizard (Step 2: Review Config)

When to Use Each Algorithm

ScenarioRecommended
Normal weekly schedulingV1 (proven, well-tested)
Nurses have very different contract hoursV2 (hour-budget-aware)
Many part-time nursesV2 (daily target adapts)
All nurses similar hours, many clinic constraintsV1 (MCV + look-ahead)
Want to compare and pick better resultRun both, compare quality scores
